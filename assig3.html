📋 Table Schema

Database Structure

We're working with 3 interconnected tables with AUTO_INCREMENT for emp_id:

  • Dept: Department information
  • Employee: Employee details with auto-generated IDs
  • Project: Project information linked to departments
USE te31145_db;

CREATE TABLE Dept (
    dept_id INT PRIMARY KEY,
    dept_name VARCHAR(50),
    dept_location VARCHAR(20)
);

CREATE TABLE Project (
    proj_id INT PRIMARY KEY,
    dept_id INT,
    proj_name VARCHAR(20),
    proj_location VARCHAR(20),
    proj_cost INT,
    proj_year INT,
    FOREIGN KEY (dept_id) REFERENCES Dept(dept_id) ON DELETE CASCADE
);

CREATE TABLE Employee (
    emp_id INT PRIMARY KEY NOT NULL AUTO_INCREMENT,
    dept_id INT,
    emp_fname VARCHAR(20),
    emp_lname VARCHAR(20),
    emp_position VARCHAR(50),
    emp_salary INT,
    emp_joinDate DATE,
    FOREIGN KEY (dept_id) REFERENCES Dept(dept_id) ON DELETE CASCADE
);

🔗 Understanding JOINs

What is a JOIN?

A JOIN combines data from multiple tables. Imagine two lists - one with student names and IDs, another with IDs and grades. JOIN helps us see "which student got which grade" by matching the IDs.

JOIN Type What It Does Simple Analogy
INNER JOIN Shows only matching records Students who submitted assignments
LEFT JOIN All from left + matching from right All students, even without submission
RIGHT JOIN All from right + matching from left All assignments, even without students
NATURAL JOIN Auto-matches common columns Smart join that figures it out

1 Employee & Department Details (NATURAL JOIN)

Advanced

💡 Concept: NATURAL JOIN

NATURAL JOIN automatically finds columns with the same name (dept_id) and joins on them. It's like a smart assistant doing the work for you - no need to specify ON conditions!

SELECT *
FROM Employee
NATURAL JOIN Dept;
Result: Shows employee details along with their department name and location, automatically matched by dept_id

2 Employee Details with Department Location

Intermediate

💡 Concept: INNER JOIN

Unlike NATURAL JOIN, here we explicitly tell which columns to match using the ON clause. This gives us more control over the join condition.

SELECT E.emp_fname, E.emp_position, D.dept_location, E.emp_joinDate
FROM Employee E
INNER JOIN Dept D ON E.dept_id = D.dept_id;
In simple words: "Show employee name, position, department location, and join date for employees who belong to a department"

3 Employees with Projects Not in Hyderabad

Advanced

💡 Concept: LEFT JOIN with Filter

LEFT JOIN keeps ALL employees, even if they don't have projects. Then the WHERE clause filters out projects in Hyderabad.

<> means "not equal to" (same as !=)

SELECT E.*, P.proj_id, P.proj_cost
FROM Employee E
LEFT JOIN Project P ON E.dept_id = P.dept_id
WHERE P.proj_location <> 'Hyderabad';
What this does: Shows all employees with their projects, excluding any projects located in Hyderabad

4 Department & Employee for 2020 Projects

Advanced

💡 Concept: Multiple JOINs & CONCAT

We can chain multiple joins to connect 3+ tables. The CONCAT() function combines first and last names into one full name column.

SELECT D.dept_name, 
       CONCAT(E.emp_fname, ' ', E.emp_lname) AS emp_name, 
       E.emp_position
FROM Employee E
RIGHT JOIN Dept D ON E.dept_id = D.dept_id
INNER JOIN Project P ON D.dept_id = P.dept_id
WHERE P.proj_year = 2020;
Result: Shows department name, full employee name, and position for all projects that started in 2020

5 Employees in High-Cost Projects

Advanced

💡 Concept: Three-Table JOIN with DISTINCT

When joining multiple tables, we might get duplicate rows. DISTINCT ensures each combination appears only once.

SELECT DISTINCT E.emp_position, D.dept_name
FROM Employee E
INNER JOIN Dept D ON E.dept_id = D.dept_id
INNER JOIN Project P ON D.dept_id = P.dept_id
WHERE P.proj_cost > 30000;
What this does: Shows unique combinations of employee positions and department names for projects costing more than 30,000

6 Projects Started in 2015

Basic

💡 Concept: Simple SELECT with WHERE

A straightforward query to filter projects by their start year.

SELECT proj_name 
FROM Project
WHERE proj_year = 2015;
Result: Shows names of all projects that started in 2015

7 Departments with Exactly 5 Employees

Advanced

💡 Concept: Subquery with GROUP BY & HAVING

GROUP BY: Groups rows with same values (like grouping students by class)

HAVING: Filters groups AFTER grouping (different from WHERE which filters rows BEFORE grouping)

COUNT(): Counts number of records in each group

SELECT D.dept_name 
FROM Dept D
WHERE dept_id IN (
    SELECT dept_id
    FROM Employee
    GROUP BY dept_id
    HAVING COUNT(emp_id) = 5
);
Step-by-step execution:
  1. Inner query groups employees by department ID
  2. Counts employees in each department
  3. HAVING filters to keep only departments with exactly 5 employees
  4. Outer query retrieves the names of those departments
💡 WHERE vs HAVING:
WHERE filters rows before grouping → "Show employees with salary > 50000"
HAVING filters groups after grouping → "Show departments with more than 5 employees"

8 Count Employees Joined Before 2009

Intermediate

💡 Concept: Aggregate Function with JOIN

Combining COUNT() with JOIN to count employees meeting specific criteria across multiple tables.

SELECT COUNT(E.emp_id) AS no_of_emp_joined_pro_before_2009
FROM Employee E
JOIN Project P ON P.dept_id = E.dept_id
WHERE YEAR(E.emp_joinDate) < 2009 AND P.proj_year < 2009;
What this counts: Employees who joined before 2009 AND are in departments with projects that started before 2009

9 Create View: Employee & Department Details

Advanced

💡 Concept: What is a VIEW?

A VIEW is like a saved query or a "virtual table". Think of it as a shortcut - instead of writing a complex query every time, create a view once and use it like a regular table.

Benefits:

  • Simplifies complex queries
  • Provides security (hide sensitive columns like salary)
  • Creates consistent interface to data
  • Reusable across multiple queries
-- Create the view
CREATE VIEW emp_dept_view AS
SELECT CONCAT(E.emp_fname, ' ', E.emp_lname) AS emp_name, 
       E.emp_position, 
       E.emp_salary, 
       E.emp_joinDate,
       D.dept_name, 
       D.dept_location
FROM Employee E, Dept D
WHERE E.dept_id = D.dept_id;

-- Use the view like a regular table
SELECT * FROM emp_dept_view;
Result: Now you can query emp_dept_view instead of writing the complex JOIN every time! The view combines employee and department data for easy access.

10 View Manipulation: DML Operations

Advanced

💡 Simple Views vs Complex Views

Simple View: Based on single table - you CAN INSERT, UPDATE, DELETE

Complex View: Based on multiple tables or has aggregate functions - usually READ-ONLY

A. Create Simple View

CREATE VIEW simple_emp_view AS
SELECT emp_id, dept_id, emp_fname, emp_lname, emp_position, emp_salary, emp_joinDate
FROM Employee;

-- Display the view
SELECT * FROM simple_emp_view;

B. INSERT through View

INSERT INTO simple_emp_view (dept_id, emp_fname, emp_lname, emp_position, emp_salary, emp_joinDate)
VALUES (1, 'Ravi', 'Sharma', 'Manager', 50000, '2023-06-01');

-- Verify insertion
SELECT * FROM simple_emp_view WHERE emp_fname = 'Ravi';
Important: The data is actually inserted into the Employee table, not the view! The view is just a window to the base table.

C. UPDATE through View

UPDATE simple_emp_view
SET emp_salary = 65000
WHERE emp_fname = 'Rahul' AND emp_lname = 'Singh';

-- Verify update
SELECT * FROM simple_emp_view WHERE emp_fname = 'Rahul';
Note: The update is reflected in the base Employee table as well

D. DELETE through View

DELETE FROM simple_emp_view
WHERE emp_fname = 'Ritu' AND emp_lname = 'Kamble';

-- Verify deletion
SELECT * FROM Employee WHERE emp_fname = 'Ritu';
Note: The record is permanently deleted from the actual Employee table

E. DROP View

DROP VIEW simple_emp_view;

-- Try to access the view (will give error)
SELECT * FROM simple_emp_view;
⚠️ Important: Dropping a view only removes the view definition, NOT the actual data in the base table! The Employee table data remains intact.

🎯 Pro Tips

1. When to use which JOIN?

• Use INNER JOIN when you only want matching records
• Use LEFT JOIN when you want all records from the first table
• Use RIGHT JOIN when you want all records from the second table
• Use NATURAL JOIN when column names are same and you want quick results

2. GROUP BY vs HAVING vs WHERE

• WHERE filters individual rows BEFORE grouping
• GROUP BY groups rows with same values
• HAVING filters groups AFTER grouping

Example: WHERE filters employees → GROUP BY counts them → HAVING filters the counts

3. When to create a VIEW?

• Complex query used repeatedly in your application
• Need to hide sensitive columns (like salary from regular users)
• Simplify database structure for end users
• Create a consistent interface even if base tables change

4. Simple vs Complex Views

Simple View (DML allowed):
• Based on single table
• No aggregate functions
• Can INSERT, UPDATE, DELETE

Complex View (Usually READ-ONLY):
• Based on multiple tables (JOINs)
• Contains aggregate functions (COUNT, AVG, etc.)
• Contains GROUP BY, DISTINCT
• Cannot perform DML operations

🔍 Common Mistakes to Avoid

1. Using WHERE instead of HAVING for grouped data
❌ Wrong: WHERE COUNT(emp_id) > 5
✅ Correct: HAVING COUNT(emp_id) > 5
2. Forgetting table aliases with multiple tables
❌ Wrong: SELECT dept_id FROM Employee JOIN Dept (ambiguous column)
✅ Correct: SELECT E.dept_id FROM Employee E JOIN Dept D
3. Trying to modify complex views
❌ Wrong: INSERT INTO view_with_joins
✅ Correct: Use simple views or modify base tables directly
4. Not understanding JOIN order
❌ Wrong: Confused about which table is "left" or "right"
✅ Correct: LEFT table = before LEFT JOIN, RIGHT table = after LEFT JOIN

💡 Important Notes on Views

View Benefits

Security: Restrict access to specific columns/rows. For example, hide salary information from regular users.

Simplicity: Simplify complex queries. Users can query a simple view instead of writing complex JOINs.

Consistency: Provide consistent interface to data. Even if base tables change, views can remain the same.

Logical Independence: Change underlying table structure without affecting application queries.
