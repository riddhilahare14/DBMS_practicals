üéØ Course Overview

What is DBMS?

A Database Management System (DBMS) is software that allows us to store, retrieve, and manage data efficiently. Think of it like a smart filing cabinet that organizes information and lets you find it quickly.

Why Learn SQL?

SQL (Structured Query Language) is the language we use to talk to databases. It's like learning English to communicate with people - SQL helps us communicate with databases to get the information we need.

Assignment 2: Basic SQL Operations

üìã Database Schema

Understanding Tables

We're working with 3 tables that are connected to each other:

  • Dept (Department): Stores information about different departments
  • Employee: Stores information about employees
  • Project: Stores information about projects
CREATE TABLE Dept (
    dept_id INT PRIMARY KEY,          -- Unique ID for each department
    dept_name VARCHAR(50),             -- Name of the department
    dept_location VARCHAR(20)          -- Where the department is located
);

CREATE TABLE Employee (
    emp_id INT PRIMARY KEY,            -- Unique ID for each employee
    dept_id INT,                       -- Which department they belong to
    emp_fname VARCHAR(20),             -- First name
    emp_lname VARCHAR(20),             -- Last name
    emp_position VARCHAR(50),          -- Their job title
    emp_salary INT,                    -- How much they earn
    emp_joinDate DATE,                 -- When they joined the company
    FOREIGN KEY (dept_id) REFERENCES Dept(dept_id)
);

Query 1: INSERT Data

Basic

üí° Concept: INSERT Statement

INSERT is like adding a new row to a spreadsheet. We tell the database which table to add data to and what values to insert.

INSERT INTO Employee VALUES
(1, 1, 'Peter', 'Parker', 'Developer', 55000, '1980-06-15'),
(2, 2, 'Harry', 'Potter', 'Analyst', 60000, '1984-07-22'),
(3, 1, 'Pamela', 'Anderson', 'Developer', 58000, '1983-04-19');
What this does: Adds 3 employees to our database with all their information.

Query 2: Filter with Multiple Conditions

Intermediate

üí° Concept: WHERE Clause with LIKE

WHERE: Filters data based on conditions (like filtering emails in your inbox)

LIKE 'p%': Finds names starting with 'p' (% means "anything can come after")

JOIN: Combines data from Employee and Dept tables

SELECT * FROM Employee e
JOIN Dept d ON d.dept_id = e.dept_id
WHERE (d.dept_name IN ('Computer', 'IT')) 
AND (e.emp_fname LIKE 'p%' OR e.emp_fname LIKE 'h%');
In simple words: "Show me all employees from Computer or IT department whose first name starts with P or H"

Query 3: Find Unique Values

Basic

üí° Concept: DISTINCT

DISTINCT removes duplicates. Like making a list of unique items from your shopping cart.

SELECT DISTINCT emp_position FROM Employee;
Result: Shows each job position only once (Developer, Analyst, HR Manager, etc.)

Query 4: Update Existing Data

Intermediate

üí° Concept: UPDATE Statement

UPDATE changes existing data. It's like editing a cell in Excel.

YEAR(): Extracts the year from a date

UPDATE Employee
SET emp_salary = emp_salary * 1.1    -- Increase salary by 10%
WHERE YEAR(emp_joinDate) < 1985;
What this does: Give a 10% raise to employees who joined before 1985

Query 5: Delete Data

Basic

üí° Concept: DELETE with CASCADE

DELETE removes rows. Because of CASCADE, when we delete a department, all employees and projects in that department are also deleted automatically.

DELETE FROM Dept
WHERE dept_location = 'Mumbai';
Warning: This deletes all departments in Mumbai AND their employees!

Query 6-7: Range Queries

Basic

üí° Concept: BETWEEN & Comparison Operators

We can find data within a range using >=, <=, or BETWEEN keyword.

-- Projects in Pune
SELECT proj_name FROM Project
WHERE proj_location = 'Pune';

-- Projects with cost between 100000 and 500000
SELECT * FROM Project
WHERE proj_cost BETWEEN 100000 AND 500000;

Query 8: Aggregate Functions

Intermediate

üí° Concept: MAX() and AVG()

MAX(): Finds the highest value (like finding the tallest person)

AVG(): Calculates average (like calculating average marks)

-- Most expensive project
SELECT * FROM Project
WHERE proj_cost = (SELECT MAX(proj_cost) FROM Project);

-- Average project cost
SELECT AVG(proj_cost) AS avg_cost FROM Project;
Subquery: The query inside parentheses runs first, then the outer query uses its result

Query 9-10: Sorting & Filtering

Basic

üí° Concept: ORDER BY and IN

ORDER BY DESC: Sorts data in descending order (Z to A)

IN: Checks if value matches any item in a list

-- Sort employees by last name (descending)
SELECT emp_id, emp_fname, emp_lname
FROM Employee
ORDER BY emp_lname DESC;

-- Projects from specific years
SELECT proj_name, proj_location, proj_cost
FROM Project
WHERE proj_year IN (2004, 2005, 2007);

Assignment 3: Advanced SQL Operations

üîó Understanding JOINS

What is a JOIN?

A JOIN combines data from multiple tables. Imagine two lists - one with student names and IDs, another with IDs and grades. JOIN helps us see "which student got which grade" by matching the IDs.

JOIN Type What It Does Simple Analogy
INNER JOIN Shows only matching records from both tables Students who submitted assignments
LEFT JOIN All records from left table + matching from right All students, even those who didn't submit
RIGHT JOIN All records from right table + matching from left All assignments, even without submissions
NATURAL JOIN Automatically matches columns with same name Smart join that figures out matching on its own

Query 1: NATURAL JOIN

Advanced

üí° Concept: Automatic Matching

NATURAL JOIN automatically finds columns with the same name (dept_id) and joins on them. It's like a smart assistant doing the work for you.

SELECT * FROM Employee
NATURAL JOIN Dept;
Result: Shows employee details along with their department name and location, automatically matched by dept_id

Query 2: INNER JOIN - Matching Records

Intermediate

üí° Concept: Explicit JOIN Condition

Unlike NATURAL JOIN, here we explicitly tell which columns to match using ON clause.

SELECT E.emp_fname, E.emp_position, D.dept_location, E.emp_joinDate
FROM Employee E
INNER JOIN Dept D ON E.dept_id = D.dept_id;
In simple words: "Show employee name, position, department location, and join date for employees who belong to a department"

Query 3: LEFT JOIN with Filter

Advanced

üí° Concept: Including Non-Matching Records

LEFT JOIN keeps ALL employees, even if they don't have projects. Then we filter out Hyderabad projects.

SELECT E.*, P.proj_id, P.proj_cost
FROM Employee E
LEFT JOIN Project P ON E.dept_id = P.dept_id
WHERE P.proj_location <> 'Hyderabad';
What this does: Shows all employees with their projects, excluding projects in Hyderabad

Query 4: RIGHT JOIN + INNER JOIN

Advanced

üí° Concept: Multiple JOINs

We can chain multiple joins to connect 3+ tables. Also using CONCAT to combine first and last names.

SELECT D.dept_name, 
       CONCAT(E.emp_fname, ' ', E.emp_lname) AS emp_name, 
       E.emp_position
FROM Employee E
RIGHT JOIN Dept D ON E.dept_id = D.dept_id
INNER JOIN Project P ON D.dept_id = P.dept_id
WHERE P.proj_year = 2020;
Result: Shows department, employee name, and position for projects started in 2020

Query 5: Three-Table JOIN with DISTINCT

Advanced

üí° Concept: Removing Duplicates from JOINs

When joining multiple tables, we might get duplicate rows. DISTINCT ensures each combination appears only once.

SELECT DISTINCT E.emp_position, D.dept_name
FROM Employee E
INNER JOIN Dept D ON E.dept_id = D.dept_id
INNER JOIN Project P ON D.dept_id = P.dept_id
WHERE P.proj_cost > 30000;

Query 7: Subquery with GROUP BY

Advanced

üí° Concept: Grouping & Counting

GROUP BY: Groups rows with same values (like grouping students by class)

HAVING: Filter groups (different from WHERE which filters rows)

SELECT D.dept_name FROM Dept D
WHERE dept_id IN (
    SELECT dept_id
    FROM Employee
    GROUP BY dept_id
    HAVING COUNT(emp_id) = 5    -- Departments with exactly 5 employees
);
Step-by-step:
  1. Inner query groups employees by department
  2. Counts employees in each department
  3. Keeps only departments with 5 employees
  4. Outer query gets names of those departments

Query 9-10: Working with VIEWS

Advanced

üí° Concept: What is a VIEW?

A VIEW is like a saved query or a "virtual table". Think of it as a shortcut - instead of writing a complex query every time, create a view once and use it like a table.

Benefits:

  • Simplifies complex queries
  • Provides security (hide sensitive columns)
  • Consistent interface to data
-- Create a view
CREATE VIEW emp_dept_view AS
SELECT CONCAT(E.emp_fname, ' ', E.emp_lname) AS emp_name, 
       E.emp_position, 
       E.emp_salary, 
       D.dept_name
FROM Employee E, Dept D
WHERE E.dept_id = D.dept_id;

-- Use the view like a table
SELECT * FROM emp_dept_view;

View Manipulation

üí° Simple Views vs Complex Views

Simple View: Based on one table - you can INSERT, UPDATE, DELETE

Complex View: Based on multiple tables or has aggregates - usually read-only

-- Create simple view
CREATE VIEW simple_emp_view AS
SELECT emp_id, dept_id, emp_fname, emp_lname, emp_salary
FROM Employee;

-- INSERT into view (affects base table)
INSERT INTO simple_emp_view VALUES 
(11, 1, 'Ravi', 'Sharma', 50000);

-- UPDATE through view
UPDATE simple_emp_view
SET emp_salary = 65000
WHERE emp_fname = 'Ravi';

-- DELETE through view
DELETE FROM simple_emp_view
WHERE emp_fname = 'Ravi';

-- DROP view (doesn't affect base table data)
DROP VIEW simple_emp_view;
Important: Changes to simple views affect the underlying table. Dropping a view only removes the view definition, not the actual data.

üéØ Pro Tips

1. When to use which JOIN?

‚Ä¢ Use INNER JOIN when you only want matching records
‚Ä¢ Use LEFT JOIN when you want all records from the first table
‚Ä¢ Use RIGHT JOIN when you want all records from the second table

2. GROUP BY vs HAVING vs WHERE

‚Ä¢ WHERE filters individual rows BEFORE grouping
‚Ä¢ HAVING filters groups AFTER grouping
‚Ä¢ Example: WHERE filters employees, HAVING filters employee counts

3. When to create a VIEW?

‚Ä¢ Complex query used repeatedly
‚Ä¢ Need to hide sensitive columns (like salary)
‚Ä¢ Simplify database structure for users
‚Ä¢ Create a consistent interface to changing data
